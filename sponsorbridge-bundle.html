# SponsorBridge — Session 1 Summary
**Date:** April 2026
**Session type:** Foundational strategy + architecture
**Status:** Pre-revenue, solo founder, no paying customers

---

## What Was Decided

### Product Identity
SponsorBridge is a purpose-built B2B SaaS platform for mid-size nonprofits
($1M–$10M budget) running formal child sponsorship programs. The founder is
an active volunteer at one such organization — the platform was born from
firsthand frustration with a fragmented stack of Excel + donor portal +
QuickBooks. This origin story is both the product requirements document and
the most defensible competitive moat.

### Core Modules (All MVP Priority)
- **SponsorPool** — full lifecycle management of the sponsor population
- **ChildRegistry** — beneficiary profiles, enrollment, sponsorship status
- **RelationshipGraph** — the live sponsor↔child matching engine, payment
  tracking, health scoring, impact reporting

### The Flagship Feature
Native QuickBooks Online integration that is bookkeeper-configured, not
developer-configured. This is the primary differentiator over every
competitor in the market. Configuration surface includes:
- OAuth connect + live chart of accounts import from QBO API
- Account mapping (5 income options, org-defined labels, QBO accounts
  pulled live — never manually typed)
- Class + program tracking (restricted/unrestricted fund tagging)
- Sync rules (real-time vs. batched, duplicate detection, donor matching)
- Pledge recognition method (on commitment vs. on cash receipt)
- Failed payment rules (hold, void, or reverse in QBO)
- Refund + reversal rules (credit memo vs. offsetting entry)
- Reconciliation method (gross revenue vs. net deposit)
- Recognition date policy (gift date vs. settlement date)
- Transaction memo templates with tokens
- Live preview panel before saving any configuration
- Full audit log exportable to CSV
- Two onboarding journeys: Net New QBO and Existing QBO

### Two QBO Onboarding Journeys
**Net New QBO** — provisions a nonprofit-ready chart of accounts in QBO
automatically. Real-time sync on by default. Auto-class creation on by
default. Generous, opinionated defaults throughout.

**Existing QBO** — pulls existing chart of accounts, classes, and customer
records via API. 30-day preview/approval queue before live sync. Batched
daily sync by default. No auto-creation of accounts or classes without
explicit permission. Conservative defaults throughout. Protects
auditor-blessed structures.

Both journeys land on the same configuration interface — only the setup
experience and defaults differ.

### Data Model Decisions (Treat as Settled)
- Single `contacts` table — donors and sponsors are not separate tables.
  Same person can wear both roles. Role/type field handles distinction.
- `sponsorships` — the relationship record (contact ↔ child). Defines
  the human relationship only, not financial behavior.
- `recurring_schedules` — financial behavior: amount, frequency, income
  account mapping, Stripe subscription ID. Income account is set at
  schedule creation time, not at the sponsorship level.
- `payments` — every transaction, referencing the schedule that generated
  it and the income account it should post to.
- `ledger_events` — QBO sync record. Today routes outward to QBO/Xero.
  Year 3 routes to native fund accounting instead. Zero schema migration
  required to make the switch.
- `programs` — org-defined program cohorts that children are assigned to
  and that map to QBO classes.

### The Unified Donation Page
A single public-facing page serving both sponsors and donors. Key decisions:
- Supports one-time and recurring payments via Stripe
- Sponsors can make additional one-time gifts on top of existing recurring
- Five dropdown options, org-defined labels, each mapped to a QBO income
  account pulled live from the org's chart of accounts
- Account names are never manually typed — always imported from QBO
- Returning visitors matched by email to existing contact records
- One-time vs. recurring is a toggle — same form, same income account
  selection, same page
- Recurring payment income account set at schedule creation, not
  at the sponsorship relationship level

### Multi-Tenancy Architecture
- Strategy: Shared DB, shared schema, Supabase RLS on every table via org_id
- RLS enforces org-level data isolation at the database layer — not
  application layer
- Child data sensitivity makes RLS security testing non-negotiable before
  launch — dedicated test suite impersonating different org JWT tokens
  required on every deploy
- Migration path to schema-per-tenant documented but not needed until
  a major enterprise customer contractually demands it

### Accounting Integration Strategy
- Integration-first, not build-your-own ledger
- Open API abstraction layer — AccountingAdapter interface that QBO, Xero,
  and future connectors plug into
- QBO and Xero as first-class connectors
- Open REST API + webhooks for all other accounting systems
- Native fund accounting is a Year 3 move — the all-in-one disruption
- Integration priority (QBO vs Xero first) to be validated by first 10
  beta customers — ask them directly, don't assume

### Tech Stack
- **Frontend:** React + TypeScript + Vite + Shadcn/UI
- **State:** Zustand + TanStack Query
- **Backend:** Node.js + Fastify + tRPC
- **Database:** Supabase (PostgreSQL + RLS)
- **Auth:** Supabase Auth + RBAC
- **Payments:** Stripe + Stripe Connect
- **QBO SDK:** node-quickbooks (abstracted behind AccountingAdapter)
- **Background jobs:** Trigger.dev
- **Email:** Resend + React Email
- **Storage:** Supabase Storage
- **Analytics:** PostHog
- **Hosting:** Vercel + Railway (~$50/mo to start)

### Pricing
- **Free:** up to 50 children / 75 sponsors — freemium funnel
- **Starter:** $199/month — basic QBO, 100 children, 3 users
- **Growth:** $499/month — unlimited, full QBO config, open API ★ sweet spot
- **Enterprise:** $999/month — SSO, custom integrations, dedicated CSM
- **Optional:** 0.5–1% platform fee on Stripe-processed payments

### Build Roadmap
- **Phase 1 (Months 1–4):** Core MVP — auth, SponsorPool, ChildRegistry,
  RelationshipGraph, manual payment entry. Validate with own org first.
- **Phase 2 (Months 5–8):** Stripe + native QBO integration + donation page
- **Phase 3 (Months 9–14):** Open API, Xero, webhooks, health scoring,
  impact reports, QBO template library
- **Phase 4 (Year 3):** Native fund accounting — the all-in-one disruption

---

## Competitive Landscape

- **REACH** — closest competitor. ~$1.6M ARR estimated. "Hundreds" of
  customers (their own language). $50–$199/month. Top tier ($199) offers
  "API Access" — a developer-facing API, not a native integration. No
  bookkeeper-configurable QBO. Serves the low end of the market.
  SponsorBridge targets orgs that have outgrown REACH or would never
  consider it.
- **Outgiven** — child privacy masking tech (Identity Mask®) is notable.
  Free/usage-based. Smaller footprint.
- **HelpYouSponsor** — lists QBO integration but unverified depth.
- **Pure Charity, Clarity** — fragmented, niche, no dominant player.
- **Bloomerang/Virtuous** — donor CRM only, no child registry, partial QBO.
- **Market conclusion:** Effectively uncaptured at the mid-size tier.
  No dominant player. REACH has mindshare but not market share.

---

## Market Sizing
- **SAM:** ~10,000 US orgs at ICP profile → ~$60M ARR domestically
- **SOM (Years 1–3):** 500 orgs × $6,000 ARR = $3M ARR
- **TAM with global expansion:** $200M–$400M ARR
- **Accounting software used by target market:** ~70–80% QuickBooks Online,
  remainder split between Xero and Sage Intacct

---

## The Three Moats
1. **Relationship Graph** — years of sponsor↔child history no org will
   migrate away from. Data moat.
2. **QBO Integration Depth** — bookkeeper-configured, not developer-configured.
   Months of careful product work that REACH has explicitly left on the table.
3. **Founder Advantage** — active volunteer at an org running this exact
   problem. Every design decision validated against lived experience.

---

## Open Questions

- Which accounting system do the first 10 beta customers actually use?
  (Validates QBO vs. Xero first — do not assume before asking)
- What are the five donation page income option labels for the founder's
  own org? (Good starting point for default template)
- Does the founder's org use QBO Classes for program tracking currently?
- What does the children tracking side look like at the founder's org —
  same Excel spreadsheet as sponsors, or separate?
- What is the exact sponsor-to-child relationship model at the founder's
  org — 1:1 dedicated, 1:many pooled, or both?
- Should the donor-facing donation page require authentication for
  returning visitors, or match by email silently after payment?
- What is HelpYouSponsor's actual QBO integration depth?
  (Needs hands-on verification — listed but unconfirmed)

## Specific Next Actions

1. **This week:** Sit with whoever manages the sponsor spreadsheet and
   whoever does QBO entries at the founder's org. Time them doing the
   work. Capture the exact workflow in steps. That session produces the
   product's value proposition in hard numbers.
2. **This week:** Sign up for REACH trial. Map every feature. Read every
   negative Capterra review. The gaps are the roadmap.
3. **Before writing schema:** Sketch the `sponsorships` and
   `recurring_schedules` tables and their relationship on paper.
   Confirm the income account lives on the schedule, not the sponsorship.
4. **Before Phase 2:** Interview first 10 beta customer orgs and ask
   directly: "What accounting system do you use?" Let the answer
   determine whether QBO or Xero integration ships first.
5. **Phase 2 design:** Build the QBO configuration page as a tabbed
   interface — one tab per configuration domain. Each tab saves
   independently. Plain-English tooltips on every setting.
6. **Phase 2 design:** Build the live preview panel for the QBO config
   page. Show a sample transaction posting to QBO based on current
   settings before anything is saved. This is the trust-builder with
   skeptical bookkeepers.

---

## Key Context for Next Session

The QBO integration configuration page is the flagship feature and the
primary sales differentiator. Every UX decision around it should be
optimized for the nonprofit bookkeeper persona — non-technical,
trust-driven, audit-conscious. Plain English over technical jargon
throughout. The bookkeeper's buy-in is the internal close.

The `ledger_events` table is the most important architectural decision
in the entire data model. It is the bridge to the all-in-one disruption
in Year 3 — today it routes outward to QBO/Xero, eventually it routes
inward to native fund accounting. Zero schema migration required.
Do not revisit this decision without a strong reason.

The founder's own nonprofit is the pilot customer zero and the reference
implementation for every feature decision. If a feature wouldn't have
saved the founder's org time last month, it doesn't ship in Phase 1.

  Tab, Student, Sponsor, Sponsorship, FinancialTransaction, 
  StudentStatus, SponsorshipStatus 
} from './types';
import { INITIAL_STUDENTS, INITIAL_SPONSORS, INITIAL_SPONSORSHIPS } from './constants';
import Sidebar from './components/Sidebar';

const STORAGE_KEY = 'legacy_africa_data';

const App: React.FC = () => {
  const [activeTab, setActiveTab] = useState<Tab>('Dashboard');
  const [currentDate, setCurrentDate] = useState<string>(new Date().toISOString().split('T')[0]);
  
  // State Initialization
  const [students, setStudents] = useState<Student[]>(() => {
    const saved = localStorage.getItem(STORAGE_KEY);
    return saved ? JSON.parse(saved).students : INITIAL_STUDENTS;
  });
  const [sponsors, setSponsors] = useState<Sponsor[]>(() => {
    const saved = localStorage.getItem(STORAGE_KEY);
    return saved ? JSON.parse(saved).sponsors : INITIAL_SPONSORS;
  });
  const [sponsorships, setSponsorships] = useState<Sponsorship[]>(() => {
    const saved = localStorage.getItem(STORAGE_KEY);
    return saved ? JSON.parse(saved).sponsorships : INITIAL_SPONSORSHIPS;
  });
  const [transactions, setTransactions] = useState<FinancialTransaction[]>(() => {
    const saved = localStorage.getItem(STORAGE_KEY);
    return saved ? JSON.parse(saved).transactions : [];
  });

  // Sync to LocalStorage
  useEffect(() => {
    localStorage.setItem(STORAGE_KEY, JSON.stringify({
      students, sponsors, sponsorships, transactions
    }));
  }, [students, sponsors, sponsorships, transactions]);

  // Utility to calculate age
  const calculateAge = useCallback((birthDate: string, referenceDate: string = currentDate) => {
    const birth = new Date(birthDate);
    const ref = new Date(referenceDate);
    let age = ref.getFullYear() - birth.getFullYear();
    const m = ref.getMonth() - birth.getMonth();
    if (m < 0 || (m === 0 && ref.getDate() < birth.getDate())) {
      age--;
    }
    return age;
  }, [currentDate]);

  // Time Simulation Engine
  const advanceTime = useCallback((months: number) => {
    const d = new Date(currentDate);
    const newDateObj = new Date(d.setMonth(d.getMonth() + months));
    const newDateStr = newDateObj.toISOString().split('T')[0];
    setCurrentDate(newDateStr);

    // 1. Update Student Statuses based on new age
    setStudents(prev => prev.map(student => {
      const age = calculateAge(student.birthDate, newDateStr);
      let newStatus = student.status;
      if (age >= 22) newStatus = StudentStatus.ALUMNI;
      else if (age >= 19) newStatus = StudentStatus.HIGHER_ED;
      else if (age >= 17 && age < 19) newStatus = StudentStatus.GRADUATING_HS;
      else if (age < 17 && student.status === StudentStatus.ALUMNI) newStatus = StudentStatus.ACTIVE; // Edge case for reverse simulation
      
      return { ...student, status: newStatus };
    }));

    // 2. Generate Contributions & Update Sponsor Lifetimes
    const newTransactions: FinancialTransaction[] = [];
    const updatedSponsorships = sponsorships.map(s => {
      if (s.status !== SponsorshipStatus.ACTIVE) return s;

      const monthlyAmount = s.monthlyPledge;
      let totalNewContribution = 0;

      for (let i = 0; i < months; i++) {
        const transDate = new Date(currentDate);
        transDate.setMonth(transDate.getMonth() + i + 1);
        const dateStr = transDate.toISOString().split('T')[0];
        
        const tx: FinancialTransaction = {
          id: `TX-${Math.random().toString(36).substr(2, 9)}`,
          type: 'Contribution',
          amount: monthlyAmount,
          date: dateStr,
          studentId: s.studentId,
          sponsorId: s.sponsorId,
          sponsorshipId: s.id
        };
        newTransactions.push(tx);
        totalNewContribution += monthlyAmount;
      }

      return {
        ...s,
        accumulatedBalance: s.accumulatedBalance + totalNewContribution
      };
    });

    // 3. Update Sponsor Lifetime Total
    setSponsors(prev => prev.map(sp => {
      const relevantTx = newTransactions.filter(tx => tx.sponsorId === sp.id);
      const added = relevantTx.reduce((sum, tx) => sum + tx.amount, 0);
      return { ...sp, lifetimeSponsored: sp.lifetimeSponsored + added };
    }));

    // 4. Quarterly Disbursements (Jan, Apr, Jul, Oct)
    // Check if we passed a quarterly boundary
    const startMonth = new Date(currentDate).getMonth();
    const endMonth = newDateObj.getMonth();
    const quarters = [0, 3, 6, 9]; // Jan, Apr, Jul, Oct
    
    let processedSponsorships = updatedSponsorships;

    for (let mOffset = 1; mOffset <= months; mOffset++) {
      const checkDate = new Date(currentDate);
      checkDate.setMonth(checkDate.getMonth() + mOffset);
      const m = checkDate.getMonth();
      
      if (quarters.includes(m)) {
        processedSponsorships = processedSponsorships.map(s => {
          if (s.accumulatedBalance > 0) {
            newTransactions.push({
              id: `TX-D-${Math.random().toString(36).substr(2, 9)}`,
              type: 'Disbursement',
              amount: s.accumulatedBalance,
              date: checkDate.toISOString().split('T')[0],
              studentId: s.studentId,
              sponsorshipId: s.id
            });
            return { ...s, accumulatedBalance: 0 };
          }
          return s;
        });
      }
    }

    setSponsorships(processedSponsorships);
    setTransactions(prev => [...prev, ...newTransactions]);
  }, [currentDate, sponsorships, calculateAge]);

  const renderContent = () => {
    switch (activeTab) {
      case 'Dashboard': return <Dashboard students={students} sponsors={sponsors} sponsorships={sponsorships} transactions={transactions} calculateAge={calculateAge} />;
      case 'Students': return <Students students={students} sponsorships={sponsorships} calculateAge={calculateAge} />;
      case 'Sponsors': return <Sponsors sponsors={sponsors} sponsorships={sponsorships} students={students} />;
      case 'Sponsorships': return <Sponsorships sponsorships={sponsorships} students={students} sponsors={sponsors} setSponsorships={setSponsorships} />;
      case 'Finances': return <Finances transactions={transactions} students={students} sponsorships={sponsorships} />;
      case 'Architecture': return <Architecture />;
      default: return <Dashboard students={students} sponsors={sponsors} sponsorships={sponsorships} transactions={transactions} calculateAge={calculateAge} />;
    }
  };

  return (
    <div className="flex h-screen bg-gray-100 overflow-hidden">
      <Sidebar 
        activeTab={activeTab} 
        setActiveTab={setActiveTab} 
        onAdvanceTime={advanceTime} 
        currentDate={currentDate}

<script>
  // Anchor TOC active state
  const sections = document.querySelectorAll('.config-section');
  const links = document.querySelectorAll('.anchor-link');
  const observer = new IntersectionObserver((entries) => {
    entries.forEach(e => {
      if (e.isIntersecting) {
        links.forEach(l => l.classList.remove('active'));
        const link = document.querySelector(`.anchor-link[href="#${e.target.id}"]`);
        if (link) link.classList.add('active');
      }
    });
  }, { rootMargin: '-100px 0px -60% 0px' });
  sections.forEach(s => observer.observe(s));

  // Toggle interactivity
  document.querySelectorAll('.toggle-track').forEach(t => {
    t.addEventListener('click', () => t.classList.toggle('on'));
  });
</script>
